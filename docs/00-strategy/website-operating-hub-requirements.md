# Website Operating Hub Requirements

## Purpose

Define the functional requirements for the Journey website and private operating hub now that Journey V3 has live Drive, Sheets, Docs and prototype surfaces. The site should feel Alpha-inspired in structure and ease of use, but all copy, resources, media, data handling and workflows must remain Journey-owned and approval-gated.

## Product Layers

1. Public explanation layer: public-safe overview of The Journey, who it serves, the rhythm, major events and how a local team starts.
2. Private leader operating layer: leader-only resources, planning templates, review packets, delegation surfaces and readiness views.
3. Management and delegation layer: owner roles, task status, launch gates, decisions, review routing, communications queues and event readiness.
4. Resource library layer: consolidated Drive folders, canonical Docs, resource index rows, quality checks and local recovery templates.
5. Review and governance layer: HR-001 through HR-007, gate evidence, access decisions, content review, leadership decision capture and publication go/no-go.
6. Build handoff layer: Website Build Backlog items BLD-001 through BLD-020, blocked until prerequisite approvals exist.

## Current Prototype Coverage

The current non-production prototype includes Dashboard, START HERE, Team, Tasks, Events, Library, Planning, Templates, Gate Review, Access, Content, Build and Drive Copy views. It uses synthetic local data and no external runtime dependencies. It is a review surface only; Drive and Sheets remain the operating source of truth.

## Data And Access Boundaries

- Public candidate content may include only approved Journey-owned explanations, high-level dates and public-safe next steps.
- Protected leader content may include templates, guide summaries, approved resource links and management summaries.
- Restricted authenticated content is required for review packets, data-owner decisions, participant administration, medical/safeguarding workflows, finance tools and unresolved gate evidence.
- Withheld content must not appear on the website until a qualified owner changes the decision in the live tracker.
- The website should link to approved Drive or Sheet surfaces instead of duplicating restricted records.

## Production Readiness Gates

- GATE-001: owner names, backups and authority are recorded before role assignment surfaces use real names.
- GATE-002: HR-001 through HR-007 are signed off by qualified reviewers before gated content appears as approved.
- GATE-003: one real leadership-team pilot is completed and accepted changes are recorded before the workflow is treated as operationally proven.
- GATE-004: restricted-data storage, access groups, field purpose and retention are approved before real local information is linked or migrated.
- GATE-005: the website access model is approved before any broader sharing or protected leader area is implemented.
- GATE-006: publication authority approves version, URL, rollback owner, validation evidence and conditions before production deployment or public sharing.

## Acceptance Criteria

- A Journey Lead can open START HERE and reach the tracker, prototype, review packets, launch gates and resource index in one path.
- Functional leads can find their templates, packets, tasks and decision surfaces without seeing restricted data outside their approved access.
- Every website section maps to a WCR row, access decision, owner role and source-of-truth surface.
- Website Build Backlog rows identify build item, owner role, required authority, status, validation and update target before technical work starts.
- Draft, review-required, blocked and approved states are visually distinct and never imply gate closure without evidence.
- No private participant, medical, finance, banking, contact or reviewer-condition data is stored in public site content.
- Validation can detect missing markers, external prototype dependencies, invalid tracked files and sensitive source-original tracking mistakes.

## Non-Approval Boundary

This requirements document does not approve website access, content, copyright use, restricted-data movement, build start, deployment, public sharing, publication, DEC-006/TASK-016 updates or GATE-006 closure. It defines the required shape of the future website once accountable human approvals exist.

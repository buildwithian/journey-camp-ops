# Leadership Review Message Templates

Role-only wording pack for the Leadership Review Communications queue. Do not store private contact details in this document or the shared tracker.

## How to Use This Pack

Choose the COMM row first.

Replace bracketed placeholders only in the sending channel.

Attach the live tracker, packet or checklist named in the COMM row.

After sending, update the COMM status and record the response or evidence link in the approved destination.

## Template 1 - Gate Alignment Meeting Invite

Use for: COMM-001.

Subject: Journey launch gate alignment review

Body:

Hello [recipient or role],

Please join the leadership review session to confirm the six open launch gates, privacy boundaries and review order.

Before the session, please review the Launch Gates, Gate Evidence Checklist, Leadership Review Schedule, Leadership Review Agenda and Leadership Review Outputs.

The meeting will not approve gates or assign real leader names. It will confirm the review sequence, evidence expectations and next owner actions.

Please bring any questions about data boundaries, decision authority or blocked prerequisites.

Response logging: Record the agreed review order and next-session owner roles in OUT-001.

## Template 2 - Owner Appointment and Governance Review Invite

Use for: COMM-002 and COMM-003.

Subject: Journey owner appointment and authority review

Body:

Hello [recipient or role],

Please join the owner appointment and authority review for the Journey operating model.

The review should confirm the authorized path for source evidence, candidate nomination, backup coverage, decision rights and escalation boundaries.

Do not copy private source names or contact details into shared records. Real names move to Leader Roster only after programme-owner confirmation and authority approval.

Response logging: Record appointment routing updates in Owner Appointment Routing and authority decisions or gaps in Governance Authority Review, DEC-003 and HR-005.

## Template 3 - Human Review Request

Use for: COMM-004 through COMM-010.

Subject: Journey human review request - [HR ID]

Body:

Hello [reviewer role],

You are being asked to review [review area] for The Journey before the related launch gate can close.

Please review the assigned packet and record reviewer authority, decision, decision date and any required conditions in Human Review.

If changes are needed, record conditions or changes required rather than approving the item.

Do not add private contact, participant, medical or financial details to the shared tracker.

Response logging: Update Human Review Routing when the request is sent and record final sign-off only in Human Review after qualified review.

## Template 4 - Pilot Planning Invite

Use for: COMM-011.

Subject: Journey leadership-team pilot planning

Body:

Hello [recipient or role],

Please join the pilot planning session to define the smallest leadership-team pilot that can test the packs, trackers and decision flow.

The session should confirm pilot scope, minimum readiness checks, observation categories and data boundaries before the pilot is run.

Do not include participant, medical, financial or private personal data in the pilot plan.

Response logging: Record the pilot scope and readiness threshold in Pilot Cycle Plan and OUT-005.

## Template 5 - Pilot Debrief Request

Use for: COMM-012.

Subject: Journey pilot debrief and change routing

Body:

Hello [recipient or role],

Now that a real leadership-team pilot has produced observations, please join the debrief to categorize findings and route changes.

Bring notes on unclear steps, duplicated work, missing resources, unnecessary fields, risks and restricted-data concerns.

Only approved process changes should update templates. Content, safety, authority and copyright items remain subject to the relevant review gates.

Response logging: Record accepted, review-required and deferred changes in Pilot Cycle Plan and OUT-006.

## Template 6 - Restricted Data Replacement Approval Request

Use for: COMM-013.

Subject: Journey restricted data replacement review

Body:

Hello [recipient or role],

Please review the proposed replacement of synthetic examples with real local information.

Confirm the accountable data owner, minimum necessary fields, storage location, access group and any information that should remain blank or withheld.

Do not migrate restricted data from this request. Migration can happen only after the approved location and access model are recorded.

Response logging: Record the data-owner approval path, required changes and storage decision in DEC-005 and the restricted data checklist.

## Template 7 - Website Access Decision Review Invite

Response logging: Record the approved access model, conditions and affected areas in Website Access Review and DEC-004.

Use for: COMM-015.

Use for: COMM-016.

Subject: Journey launch gate status digest

Body:

Hello team,

This is the weekly role-only status digest for the remaining Journey launch gates.

Open counts to check: launch gates, owner evidence reviews, website access decisions, governance authority decisions, human review routing, owner appointment routing, pilot cycle plan, gate evidence, leadership review sessions, outputs, agenda items and communications.

Next actions: [list the next three COMM rows needing action].

Do not include private contact details, participant information, medical details, financial details or unapproved source content in this digest.

Response logging: Update the relevant COMM rows, Launch Gates evidence links and downstream tracker rows after replies are received.

## Status Discipline

Use `Draft needed` when the sender still needs to compose the message.

Use `Ready to send` only after required links or attachments are confirmed.

Use `Sent - awaiting reply` only after the authorized sender sends the message outside this shared tracker.

Use `Reply received` when a response exists but the proper tracker or evidence destination is not yet updated.

Use `Action logged` only after the response has been recorded in the approved destination.

Use `Blocked` when a prerequisite such as pilot evidence, access approval or review authority is missing.

Use `Not needed` only when a human owner decides the communication is unnecessary and records why.

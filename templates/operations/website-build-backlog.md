# Website Build Backlog

## Purpose

Use this backlog to prepare the production website build after accountable leaders approve the required access, content, copyright, data, authority and publication prerequisites. It translates the desired Alpha-style operating hub into concrete Journey build items while keeping Drive and Sheets as the source of truth.

## Current Live-state Note

The live `Website Build Backlog` tab is populated with the 20 rows from [website-build-backlog.csv](website-build-backlog.csv), linked into Dashboard row 25, Instructions row 33 and Resource Library Index quality checks. Dashboard row 25 currently shows 20 blocked or pending build items.

## Privacy And Approval Rule

Keep `Approved scope / decision ref`, `Target environment / URL` and `Evidence / artifact link` blank until the accountable authority records real approval evidence. Do not store restricted data, private names, participant details, medical information, banking details, unpublished review evidence or credentials in this backlog.

## Status Values

- Blocked until prerequisites accepted
- Needs access approval
- Needs content approval
- Needs copyright approval
- Needs data owner approval
- Needs authority approval
- Needs reviewer decision
- Needs pilot run
- Needs build scope
- Ready for build after approval
- Build in progress
- Built for review
- Changes required
- Accepted for publication review
- Not approved / withheld

## Backlog Coverage

The CSV template contains 20 build-readiness rows:

- BLD-001 through BLD-003 for source-of-truth boundaries, information architecture and approved original content.
- BLD-004 through BLD-008 for public overview pages, explanation videos, dates, leader resource navigation and planning templates.
- BLD-009 through BLD-012 for leader assignment surfaces, management dashboards, reviewer routing and pilot evidence summaries.
- BLD-013 through BLD-015 for restricted-data segregation, authentication/access controls and form/intake boundaries.
- BLD-016 through BLD-020 for QA, rollback, deployment preparation, publication package and post-launch management.

## Operating Sequence

1. Confirm `Leadership Decision Capture`, `Website Access Review`, `Website Content Review`, HR-007, DEC-004 and DEC-006 evidence before moving any row out of a pending or blocked status.
2. Keep build scope blank until the relevant approval surface identifies the content, access model, owner, evidence link and conditions.
3. Use local prototype files only as design and workflow references until publication authority approves a production build.
4. Update DEC-006, TASK-016, Event Readiness and Launch Gates only after the publication authority accepts the relevant build evidence.
5. Keep Drive and Sheets as the operating source of truth; the website must link to approved source surfaces rather than duplicate restricted data.

## Non-Approval Boundary

This backlog does not approve website access, content, copyright use, restricted-data movement, technical build start, deployment, public sharing, publication, DEC-006/TASK-016 updates or GATE-006 closure. It makes future build work executable after leaders approve it.
